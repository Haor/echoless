# 延迟 Probe 质量判定改进计划

状态：待实施

适用范围：Echoless 1.2.x 后续开发

## 问题结论

当前 Probe 的 `stable` 只表示已选延迟值的标准差和首尾漂移较小，不能证明测量本身有效。
麦克风静音、低电平底噪、环境讲话或错误路由都可能产生稳定但错误的结果，并被自动写入
`near_delay_ms` 或 `initial_delay_ms`。

这是必须修复的正确性问题。Probe 的假阴性只会要求用户重试，假阳性却会把错误参数写入
音频管线，持续影响回声消除，因此质量门应当偏保守。

## 当前实现

实现位于 `crates/echoless-cli/src/probe_delay.rs` 和
`app/src/pages/AdvancedPage.tsx`：

1. CLI 生成 12 次、每次 70 毫秒的蜂鸣，频率按 880、1320、1760、1100 Hz 循环。
2. 参考和麦克风录音被压缩成 0.5 毫秒 RMS 能量包络，频率身份在分析阶段被丢弃。
3. 参考包络通过动态能量阈值寻找事件，每个事件在正负 250 毫秒内搜索最大绝对相关性。
4. `|corr| > 0.15` 的事件进入延迟集合，负相关也会被接受。
5. GUI 在没有 warning、延迟标准差小于 5 毫秒且首尾漂移小于 10 毫秒时显示 `stable`。
6. macOS 收到成功结果后直接写入建议近端延迟；Windows 只在上述 `stable` 条件成立时写入
   AEC3 初始延迟。

## 已确认缺陷

### 1. 空集合被判为稳定

没有任何事件通过相关性门槛时，CLI 会退回全局相关结果，并把标准差和漂移都写成 0。
GUI 因而把一次没有有效事件的测量判为稳定。

### 2. 单个事件天然稳定

只有一个偶然事件通过门槛时，标准差和漂移同样都是 0。当前实现不要求最低有效事件数，
也不检查有效事件占 12 次蜂鸣的比例。

### 3. 麦克风静音不产生 warning

CLI 只在 reference 低于 -45 dBFS 时报告 warning，没有检查 mic 电平。GUI 虽然会把低于
-55 dBFS 的 mic 显示为 `NO SIGNAL`，但这个显示条件没有参与 `stable` 或自动回填。

### 4. 环境噪声容易产生偶然峰值

每个事件会扫描约 1001 个候选延迟。固定的 0.15 相关性门槛没有补偿多候选搜索带来的
偶然最大值，且绝对值比较会接受负相关。一个或少量噪声峰值因此可能形成“稳定”结果。

### 5. 能量包络没有验证蜂鸣身份

已生成的四频蜂鸣序列在分析时被压成普通能量脉冲。讲话、敲击或其他系统声音只要形成
相似的能量起伏，就可能被当成探测信号。

### 6. 自动回填标准跨平台不一致

macOS 的 `near_delay_ms` 回填没有经过 GUI 的稳定门；Windows 的门虽然存在，但缺少信号、
事件数和相关性检查。CLI、GUI 展示和回填逻辑没有共享一个权威质量结论。

### 7. 缺少失败场景测试

现有算法测试只覆盖理想脉冲和已知延迟，没有覆盖数字静音、低电平噪声、环境讲话、错误
路由、单事件、少量有效事件、负相关或搜索边界命中。

## 改进原则

- 后端负责产生唯一的质量结论，GUI 不再根据零散字段自行推导 `stable`。
- 测量有效性与测量一致性分开判断。先证明信号可信，再评价延迟是否稳定。
- 无法证明有效时不得自动回填，但仍可保留诊断字段供维护者分析。
- `global_lag_ms` 只作为诊断信息，不再作为零有效事件时的回填候选。
- macOS 与 Windows 使用同一质量门；平台差异只保留在最终参数映射中。
- 阈值必须通过合成样本和真实录音校准，不能仅凭单次设备实测写死。

## 第一阶段：先阻止错误回填

第一阶段保留现有包络相关算法，只增加保守的质量门和统一契约。

### 后端质量结果

为 Probe 结果增加权威质量字段，例如：

```text
quality: valid | uncertain | invalid
quality_reasons:
  - ref_signal_missing
  - mic_signal_missing
  - insufficient_reference_events
  - insufficient_valid_lags
  - weak_correlation
  - inconsistent_lags
  - lag_at_search_boundary
```

具体命名可在实现时调整，但 CLI 与 GUI 必须消费同一结果。

### 必须执行的硬门

1. reference 和 mic 都必须有可用信号。绝对 dBFS 只能作为兜底，优先比较蜂鸣时段与前导
   静音时段的能量差，避免高底噪被误认为有效信号。
2. reference 必须检测到足够多的预期蜂鸣，且有效延迟事件必须达到最低数量和最低比例。
3. 相关性必须为正，并达到经过样本校准的门槛；不再使用 `abs(corr)` 接受负相关。
4. 零个或一个有效事件不能产生 `valid` 结果，也不能使用全局相关结果回填。
5. 延迟集中度使用稳健统计量判断。建议以中位数作为最终延迟，并用 MAD 或内点范围过滤
   离群事件，再检查内点数量、标准差和漂移。
6. 结果命中正负 250 毫秒搜索边界时判为不可信，因为真实峰值可能在搜索窗外。

首轮实验可从“12 次蜂鸣中至少 8 次形成同一延迟簇”开始，但这只是待校准候选值，不作为
文档预先锁死的产品契约。

### 回填纪律

- 只有 `quality == valid` 才允许写入 `near_delay_ms` 或 `initial_delay_ms`。
- `uncertain` 和 `invalid` 只显示原因并建议重试，不修改现有配置。
- GUI 的 `stable` 文案应直接来自后端质量结果；`Ref OK`、`Mic OK` 与最终状态必须一致。
- AEC3 初始延迟和 macOS 近端延迟必须在同一个质量门之后执行。

## 第二阶段：提高嘈杂环境识别能力

如果第一阶段在真实噪声样本中仍有较高误判，再引入频率感知分析：

1. 利用现有 880、1320、1760、1100 Hz 循环序列验证每个事件的频率身份，而不是只匹配
   宽带能量包络。
2. 可选择对每个频率做窄带能量或匹配滤波，再按已知序列组合事件置信度。
3. 参考和麦克风必须在相同序号上识别到对应频率，普通讲话或随机敲击不能仅凭能量峰通过。
4. 保留 0.5 毫秒延迟分辨率，但将候选搜索限制在通过身份验证的事件上。

这一阶段不应先于第一阶段质量门实施。现有算法只要拒绝不可信结果，就已经能消除错误自动
回填的主要产品风险。

## 测试计划

### 确定性合成测试

- reference 正常、mic 全零：必须 invalid，不回填。
- reference 正常、mic 仅低电平白噪声：必须 invalid。
- reference 正常、mic 只有一个匹配脉冲：不得 valid。
- reference 只检测到少量蜂鸣：不得 valid。
- 负相关峰高于正相关峰：不得把负相关作为有效事件。
- 最佳延迟等于正负 250 毫秒边界：必须 uncertain 或 invalid。
- 多数事件一致、少量离群：稳健中位数应保持正确，离群事件不得拉偏建议值。
- 已知正负延迟并叠加不同强度噪声：在规定信噪比范围内保持正确。

### 真实录音测试

- 麦克风硬件静音和软件静音。
- 安静房间、风扇持续噪声、键盘敲击、背景讲话和近端主动说话。
- reference 路由错误、蜂鸣播到非参考输出、麦克风选择错误。
- Windows WASAPI loopback 与 macOS Process Tap 各保留至少一组有效和无效样本。

真实录音夹具如果包含设备信息或体积过大，可保留在 Git 忽略的本地审计目录；仓库测试应
提交可复现的最小派生样本或合成夹具。

## 实施顺序

1. 为当前误判建立静音、噪声和单事件的失败测试。
2. 在 CLI 中实现统一质量判定，删除零事件时可回填的全局 fallback。
3. 用稳健延迟簇替代直接平均，并补充事件数、相关性和边界门。
4. 扩展 JSON 契约和文档，让 CLI 暴露质量状态与原因。
5. GUI 改为消费后端质量状态，统一 stable 文案和跨平台回填条件。
6. 使用合成样本和 Windows/macOS 真实录音校准候选阈值。
7. 只有第一阶段仍不能可靠区分环境噪声时，才实施频率感知匹配。

## 验收标准

- 静音、错误路由、少量事件和纯噪声不会显示 stable，也不会修改配置。
- 有效 Probe 在受支持平台仍能给出可重复的延迟建议。
- CLI、GUI 状态文案和自动回填使用同一质量结论。
- 无有效事件时不会用全局相关结果伪造零标准差、零漂移的稳定结果。
- 回填前的所有质量门都有对应回归测试。

## 不在本轮范围

- 改写设备采集、系统音频参考或虚拟音频路由。
- 为了保留旧错误行为而增加兼容分支。
- 在没有样本证据前承诺固定的事件数、相关性或信噪比阈值。
- 把 Linux 硬件验证设为该改进的发布阻塞项。
