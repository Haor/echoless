# 通用降噪节点与模型能力分离规格

状态：本地规格草案，待拆分任务

适用范围：Echoless 1.1.0 之后的开发版本

## 问题陈述

Echoless 当前把“回声消除模型”和“背景噪声抑制”混合在同一套产品状态中。主页面的
NOISE `ON/OFF` 在不同模型下表达不同含义：AEC3 切换内置 WebRTC NS，LocalVQE 切换
模型版本，NVAFX 则没有同等的二级降噪选择。用户无法独立决定使用哪个 AEC 模型和哪个
后置降噪器，也无法从界面准确理解 LocalVQE 各版本是否已经包含噪声抑制。

这种耦合还会带来正确性风险：如果通用降噪直接叠加到包含内置 NS 的 LocalVQE 1.2 或
1.3，处理链会发生双重降噪；如果只在前端禁用选项，旧配置、CLI 配置或切换竞态仍可能
绕过限制。AEC3 的私有 `ns` 状态也会继续成为第二套真理来源，使 UI、配置和实际处理链
难以保持一致。

从产品角度看，用户需要的是一个可预测的二维选择：MODEL 只决定回声消除能力，NS 只
决定是否在 AEC 输出后加载一个通用噪声抑制节点。界面、配置、运行时处理链和诊断信息
必须对这两个选择给出一致答案。

## 解决方案

将模型选择和二级降噪拆成两个独立产品维度：

```text
MODEL: AEC3 | LVQE | NVAFX
NS:    WEBRTC | RNNOISE | OFF
```

MODEL 负责使用参考信号进行声学回声消除。NS 是互斥的后处理选择：

- `WEBRTC` 加载通用 WebRTC 降噪节点；
- `RNNOISE` 加载通用 RNNoise 节点；
- `OFF` 不加载外接 NS 节点。

AEC3、NVAFX 和纯 AEC 的 LocalVQE 1.4 可以使用全部三种 NS 选择。包含内置 NS 的
LocalVQE 1.2 与 1.3 必须强制使用 `OFF`，并在前端禁用另外两个选项。该约束同时存在于
前端状态规范化和后端配置验证中，保证 UI、CLI、手工配置和重启竞态都不能构造双重 NS。

WebRTC NS 从 AEC3 内部拆成真正的通用处理器节点。AEC3 产品路径始终关闭内部 NS；
选择 `WEBRTC` 时，实际处理链为 AEC3 后接 WebRTC NS。拆分模拟、AECMOS 和用户盲听没有
发现一致性质量回退，因此不保留旧集成路径、兼容开关或隐藏回退选项。

主页面的 NOISE `ON/OFF` 改为与 MODEL 同构的三段式 `WEBRTC / RNNOISE / OFF` 控件。
LocalVQE 模型用能力标签和说明文案表达内置能力，而不是继续用 `STD` 或文件路径暗示差异。

## 用户故事

1. 作为 Echoless 用户，我希望分别选择 AEC 模型和降噪器，从而自由组合自己偏好的回声消除与背景降噪效果。
2. 作为 AEC3 用户，我希望选择 WebRTC NS、RNNoise 或不使用外接 NS，从而避免 AEC3 被固定绑定到一种内置降噪器。
3. 作为 NVAFX 用户，我希望获得与 AEC3 相同的外接 NS 选择，从而无需更换 AEC 引擎就能处理残留背景噪声。
4. 作为 LocalVQE 1.4 用户，我希望使用全部外接 NS 选项，从而让纯 AEC 模型能够搭配独立降噪器。
5. 作为 LocalVQE 1.2 用户，我希望外接 NS 选项自动禁用，从而避免模型内置 NS 被重复执行。
6. 作为 LocalVQE 1.3 用户，我希望外接 NS 选项自动禁用，从而避免模型内置 NS 被重复执行。
7. 作为切换到 LocalVQE 1.2 或 1.3 的用户，我希望 NS 原子地切换为 `OFF`，从而避免短暂启动非法的双重 NS 处理链。
8. 作为从 LocalVQE 1.2 或 1.3 切换到其他模型的用户，我希望 NS 保持 `OFF`，从而避免应用悄悄恢复我没有明确选择的处理。
9. 作为用户，我希望当前 NS 选项明显高亮，从而一眼确认正在运行的音频管线。
10. 作为用户，我希望不可用的 NS 选项明显置灰，从而在点击前理解模型的能力限制。
11. 作为使用键盘或辅助技术的用户，我希望被禁用的 NS 选项同时拒绝键盘和指针操作，从而保证控件行为一致且可访问。
12. 作为用户，我希望再次点击当前 NS 选项时不发生任何操作，从而避免误触导致处理链重载或音频中断。
13. 作为用户，我希望任何时刻只能启用一个 NS，从而避免应用意外串联 WebRTC NS 与 RNNoise。
14. 作为 LocalVQE 用户，我希望 v1.2 和 v1.3 标记为带 NS 的模型，从而明确它们已经包含降噪。
15. 作为 LocalVQE 用户，我希望 v1.4 标记为纯 AEC 模型，从而理解它为什么可以使用外接 NS。
16. 作为 LocalVQE 用户，我希望模型悬停说明解释模型行为，而不是显示文件系统路径，从而帮助我做出选择。
17. 作为中文用户，我希望能力标签和说明使用一致的中文，从而清楚理解模型与 NS 的关系。
18. 作为英文界面用户，我希望能力标签和说明保持一致翻译，从而清楚理解模型与 NS 的关系。
19. 作为 CLI 用户，我希望在没有 GUI 的情况下也执行相同的模型与 NS 兼容规则，从而避免手写配置生成不受支持的处理链。
20. 作为 CLI 用户，我希望非法双重 NS 配置返回明确错误，从而可以修正配置，而不是获得含糊的音频行为。
21. 作为加载旧配置或外部编辑配置的用户，我希望运行前规范化或拒绝非法组合，从而避免界面状态与实际处理链不一致。
22. 作为在运行期间切换模型的用户，我希望新处理链启动前移除旧 NS 节点，从而避免重启后遗留过期节点。
23. 作为在运行期间切换 NS 的用户，我希望新处理链只包含明确选择的降噪器，从而让实际听感与界面一致。
24. 作为用户，我希望 NS 切换失败通过现有运行失败机制显示，从而避免应用在加载失败时仍宣称降噪器已启用。
25. 作为诊断功能用户，我希望处理器统计显示当前 NS 节点名称，从而确认实际运行了哪种处理。
26. 作为诊断功能用户，我希望看到 NS 处理耗时和错误，从而把性能问题或运行故障归因到正确节点。
27. 作为诊断功能用户，我希望 NS 延迟与 AEC 延迟分别上报，从而让管线延迟显示符合实际选择。
28. 作为用户，我希望 WebRTC NS 保留产品已有的默认强度，从而避免拆分架构时顺带引入无关的调音变化。
29. 作为用户，我希望 WebRTC NS 节点上报经过测量的额定延迟，从而让架构变化后的音画和语音时序仍可解释。
30. 作为用户，我希望 NS 节点保持帧长度并输出有限数值，从而避免启用降噪后污染或破坏输出设备。
31. 作为长时间使用的用户，我希望 NS 实时处理不分配内存，从而避免降噪带来周期性卡顿或爆音。
32. 作为重启或重置管线的用户，我希望每个 NS 节点可以稳定重置和预热，从而避免旧状态泄漏到新一轮运行。
33. 作为维护者，我希望所有产品路径中的 AEC3 都关闭内部 NS，从而只保留一个外接降噪真理来源。
34. 作为维护者，我希望能力矩阵只定义一次并供界面和配置逻辑共同使用，从而避免前端标签与后端合法性独立漂移。
35. 作为维护者，我希望处理器清单描述新的 NS 节点，从而让 CLI 查询、GUI 配置、诊断与测试共享同一份能力声明。
36. 作为发布维护者，我希望在受支持平台验证 WebRTC NS 和 RNNoise 的打包结果，从而保证 GUI 展示的选项在发行包中确实可以加载。
37. 作为 Windows 用户，我希望该功能继续使用现有 VB-CABLE 端点方案，从而无需为 NS 架构变化安装新驱动。
38. 作为 macOS 用户，我希望该功能继续使用现有 BlackHole 端点方案，从而不改变系统音频采集和路由方式。

## 实现决策

- 产品状态拆分为引擎选择和包含三个取值的降噪模式。AEC3 私有的布尔型 NS 开关不再代表产品级降噪状态。
- 降噪模式只有三个互斥取值：WebRTC、RNNoise 和 `OFF`。`OFF` 表示不加载任何外接 NS 处理器节点。
- AEC3、NVAFX 和 LocalVQE 1.4 允许全部三种降噪模式。LocalVQE 1.2 和 1.3 因为已经包含内置 NS，只允许 `OFF`。
- 选择 LocalVQE 1.2 或 1.3 时，必须原子地把降噪模式规范化为 `OFF`，从目标处理链移除外接 NS 节点，并在界面中禁用 WebRTC 和 RNNoise。
- 离开带内置 NS 的 LocalVQE 模型时不恢复先前的降噪模式。选择兼容模型或引擎后，用户必须再次明确选择外接 NS。
- 兼容性执行两层约束：前端规范化负责即时、可理解的交互，后端验证负责阻止手工配置、旧配置或竞态绕过规则。
- 产品处理链把 `OFF` 映射为单独的 AEC 引擎，把 WebRTC 映射为 AEC 引擎后接一个 WebRTC NS 节点，把 RNNoise 映射为 AEC 引擎后接一个 RNNoise 节点。
- 所有产品处理链中的 AEC3 都关闭内部 NS。WebRTC 选项只通过共享的 AEC 后置处理器节点实现。
- 删除旧的 AEC3 集成式 NS 产品路径。不保留兼容开关、隐藏回退或并行实现，也不要求兼容旧的 AEC3 私有 NS 配置。
- WebRTC NS 使用只启用 NS 的音频处理模块配置实现，不直接调用底层抑制器。这样可以在模块边界内保留现有的频带拆分、上频带处理、限幅和帧状态管理。
- WebRTC 节点沿用产品当前的默认抑制强度。重新设计强度选择不属于主页面改造；如果保留高级设置，它必须配置 WebRTC 节点而不是 AEC3。
- WebRTC 节点上报 6.5 毫秒额定算法延迟。离线测试锁定这一契约；后续依赖变化意外改变延迟时必须失败。
- RNNoise 在原生 48 kHz、单声道、10 毫秒处理域运行。不要求从零用 Rust 重写 RNNoise 算法。最终选择的维护中绑定或运行时必须满足许可、受支持平台打包、实时行为和共享节点契约。
- 两个 NS 处理器都实现现有处理器接口，接收前一个引擎的近端输出帧。处理链仍会传入远端参考，但两个 NS 节点都不读取它。
- 两个 NS 节点都支持重置、可预测预热、非有限采样防御、处理耗时统计、错误上报和延迟上报。它们不是回声消除器，因此不报告 ERLE。
- 主页面控件是与 MODEL 视觉对齐的三段式选择器。当前项高亮；禁用项忽略指针、触摸和键盘确认；再次激活当前项不执行任何操作。
- LocalVQE v1.2 和 v1.3 显示 `NS` 能力标签，v1.4 显示 `AEC` 能力标签，删除现有的 `STD` 标签。
- 模型悬停说明用于解释模型是“AEC 加内置 NS”还是“纯 AEC”。模型文件路径仍属于诊断信息，不作为主要悬停文案。
- 处理器发现清单把 WebRTC NS 和 RNNoise 声明为真正节点。配置生成和诊断使用这份声明，不再维护只存在于 GUI 的处理器列表。
- 模型或 NS 切换使用现有生命周期纪律，使过期的排队中或执行中的重启任务失效，防止旧处理链在用户做出新选择后重新生效。
- 本次架构变化不修改参考音频采集、麦克风采集、输出路由、时钟匹配、延迟探测或虚拟设备选择。

## 测试决策

- 测试验证外部可见的处理链和产品状态行为，不绑定私有辅助函数的内部结构。只要合法处理链、输出行为、延迟契约和界面转换保持不变，重构就不应要求重写测试。
- 后端主测试边界从产品配置开始，构造完整处理链并处理确定性的 48 kHz、单声道、10 毫秒帧，验证每一种合法引擎、模型和降噪模式组合实际生成的处理链。
- 后端矩阵测试验证 AEC3、NVAFX 和 LocalVQE 1.4 接受 WebRTC、RNNoise 和 `OFF`；LocalVQE 1.2/1.3 必须在运行开始前拒绝或规范化所有外接 NS 组合。
- 后端测试验证 WebRTC 选择生成一个 AEC 节点并且只后接一个 WebRTC NS 节点；RNNoise 选择只后接一个 RNNoise 节点；`OFF` 不生成外接 NS 节点。
- AEC3 测试验证迁移后内部 NS 始终关闭，并且无法通过产品配置路径重新开启。
- 离线音频测试验证输入输出帧数相同、输出全部为有限数值、不产生持续异常静音、不因非法值引入削波，并验证两个 NS 节点可以正确重置。
- WebRTC 离线延迟测试锁定 6.5 毫秒额定延迟契约。已有的拆分与集成实验只作为支持证据，不要求把实验数据作为测试夹具提交到 Git。
- 扩展现有处理链内存分配测试，证明新的 NS 节点在稳定实时处理阶段不会分配内存。
- 处理器统计测试验证节点身份、处理耗时、错误传播和延迟，并验证 NS 节点不会伪造回声消除指标。
- 前端测试边界是纯粹的引擎、模型和降噪模式规范化行为，沿用现有引擎逻辑测试风格。测试覆盖能力可用性、强制 `OFF`、不自动恢复以及重复选择当前项时不执行操作。
- 前端交互测试验证禁用项无法通过指针或键盘激活，并验证各 LocalVQE 模型的标签和悬停说明符合声明能力。
- 运行时与生命周期测试验证切换模型或 NS 时，旧重启任务无法激活过期处理链。
- 优先复用现有处理链测试、内存分配测试、引擎逻辑测试、运行时控制测试和运行事件测试。只有这些测试无法观察所需外部行为时才增加新边界。
- 发布验证包括 Rust 根工作区测试和 Clippy、独立 AEC3 工作区测试、前端类型检查和 Vitest、Tauri 后端测试以及受支持平台安装包冒烟测试。
- Windows 和 macOS 安装包运行验证与本地代码验证、CI 验证相互独立。除非另行要求，Linux 硬件验证不作为本功能的发布门槛。

## 不在本轮范围

- 用 WebRTC NS 或 RNNoise 替代依赖参考信号的 AEC。
- 把任一 NS 后端定义为残余回声消除器。
- 同时运行 WebRTC NS 和 RNNoise。
- 允许 LocalVQE 1.2 或 1.3 关闭模型内置降噪。
- 保留旧的 AEC3 集成式 NS 路径、兼容开关，或为旧 AEC3 私有 NS 配置提供自动迁移。
- 从零用 Rust 重写 RNNoise 算法。
- 重新设计麦克风采集、系统音频参考采集、输出设备、虚拟音频路由、时钟匹配或延迟探测。
- 新增原生虚拟麦克风或 Echoless 音频驱动。
- 恢复自定义运行时、模型或诊断目录。
- 把付费代码签名或公证设为发布要求。
- 把 Linux 硬件验证设为发布阻塞项。
- 把模型文件系统路径作为模型选择的主要帮助文案。
- 离开 LocalVQE 1.2 或 1.3 后自动恢复之前选择的 NS 模式。
- 把本地模拟程序、AECMOS 环境、源数据集音频或盲听样本提交到产品仓库。

## 补充说明

- 架构与产品行为来自现有的通用 NS 架构设计文档。该文档记录了生成本规格所依据的详细状态矩阵、节点契约、实验方法和实现顺序。
- WebRTC NS 拆分实验使用 AEC Challenge 派生便利样本，覆盖远端单讲、近端单讲和双讲。实验没有发现非有限数值、削波、输出长度变化或方向一致的 AECMOS 回退。
- 用户盲听认为集成管线与拆分管线几乎没有区别，并在双讲场景略微偏好拆分管线。这一结果构成删除 AEC3 集成式 NS 路径的产品裁决依据。
- 本地实验产物按约定保留在 Git 忽略的 `audit` 工作区。产品测试应编码实验得出的行为契约，不依赖这些本地文件。
- RNNoise 使用固定版本的 Xiph 官方运行时和官方完整模型,不采用停留在旧网络代际的 Rust 移植。构建静态编入源码与模型,运行时不下载模型、不依赖外部动态库;版本、模型哈希、许可和更新来源必须留档。
- 本草案尚未发布到 GitHub。批准后应使用 `/to-tickets` 将其拆成可独立验证、带明确阻塞关系的纵向任务。
